# Implementation Plan: Gym API & UI Integration

## Overview

Implement the complete production-ready gym management dashboard by building on the existing skeleton. Work is ordered so that foundational infrastructure (apiClient fixes, shared types, DataTable enhancements, new shared components) is completed before module-specific pages. Each CRUD module follows the same fetch → DataTable → Modal → ConfirmDialog → toast → refresh cycle. Property-based tests using `fast-check` are placed immediately after the logic they validate. Run tests with `vitest --run`.

## Notes

- Tasks marked with `*` are optional and can be skipped for a faster MVP
- Each task references specific requirements for traceability
- All API calls must go through `apiClient` — no raw `fetch` in page or modal components
- Property tests use `fast-check` (already installed) and run via `vitest --run`
- The 14 correctness properties from the design document are covered by PBT sub-tasks: Properties 1–9 and 12–14 in tasks 1, 2, 6, 12, 13, 14; Properties 10–11 are already covered by the existing `PaginationControls.test.tsx`
- Checkpoints at tasks 4, 7, 11, and 16 ensure incremental validation
- New pages (subscriptions, branches, permissions) must be wrapped with `PermissionGuard` before rendering any content
